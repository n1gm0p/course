## ГЛОССАРИЙ

**API (Application Programming Interface)** — программный интерфейс приложения; набор правил и методов, по которым клиентское приложение обращается к серверу.

**JWT (JSON Web Token)** — стандарт токена для аутентификации; после входа сервер выдаёт токен, который клиент передаёт в заголовке запроса.

**MVP (Minimum Viable Product)** — минимально жизнеспособный продукт; версия системы с базовым набором функций для проверки идеи.

**React** — библиотека JavaScript для построения пользовательского интерфейса на основе компонентов и виртуального DOM.

**REST** — архитектурный стиль построения веб-сервисов; взаимодействие по HTTP с ресурсами (GET, POST, PUT, DELETE).

**Sequelize** — ORM для Node.js, позволяющая работать с базой данных (в том числе PostgreSQL) через объекты и модели.

**Vite** — инструмент сборки фронтенда; обеспечивает быструую разработку и сборку проектов на React/TypeScript.

**Модерация** — проверка контента (мест, комментариев) перед публикацией; выполняется администратором.

**СУБД** — система управления базами данных (в проекте — PostgreSQL).

**Фронтенд** — клиентская часть информационной системы (интерфейс в браузере).

**Бэкенд** — серверная часть информационной системы (API, база данных, бизнес-логика).

## ГЛАВА 1. ПРОЕКТИРОВАНИЕ ИНФОРМАЦИОННОЙ СИСТЕМЫ

### 1.1 Анализ предметной области

Предметная область — каталог интересных мест города Элисты с возможностью участия пользователей. Пользователи могут просматривать места по категориям (достопримечательности, парки, еда, культура и др.), искать по названию и автору, добавлять места в избранное. Зарегистрированные пользователи могут создавать новые места (название, описание, категория, фотографии), редактировать и удалять свои места, оставлять комментарии с возможностью прикрепления фотографий. Для исключения неуместного контента введена модерация: новые места и комментарии проходят проверку администратором и отображаются в общем каталоге только после одобрения. Администратор может одобрять, отклонять (с указанием причины) и удалять контент. Дополнительно реализованы: уникальное имя пользователя (никнейм), восстановление пароля по email, рейтинг мест в виде «лотосов» (1–5), загрузка фотографий на сервер.

Выделены основные роли: гость (просмотр одобренного каталога), зарегистрированный пользователь (добавление мест и комментариев, избранное, профиль), администратор (модерация, удаление чужого контента). Функциональные требования включают регистрацию и вход по email и паролю, JWT-аутентификацию, CRUD для мест (с учётом прав автора и модерации), комментарии с фото, избранное, рейтинг, загрузку файлов, фильтрацию и поиск.

### 1.2 Выбор средств реализации

Для достижения цели курсовой работы выбраны следующие средства.

**Серверная часть:** Node.js и фреймворк Express. Язык — JavaScript (среда выполнения Node.js). Выбор обоснован распространённостью стека, быстрой разработкой REST API и удобной интеграцией с базами данных через драйверы и ORM.

**База данных:** СУБД PostgreSQL. Хранение пользователей, категорий, мест, фотографий мест, комментариев и фотографий комментариев, избранного, рейтингов. Связи между сущностями реализованы через внешние ключи; для доступа к данным используется ORM Sequelize, что упрощает миграции и поддержку кода.

**Клиентская часть:** библиотека React и язык TypeScript. Сборка и сервер разработки — Vite. Выбор обоснован компонентной архитектурой, типизацией (TypeScript), богатой экосистемой и возможностью построения одностраничного приложения (SPA) с маршрутизацией (React Router).

**Аутентификация:** JWT (JSON Web Token). После успешного входа сервер возвращает токен; клиент сохраняет его и передаёт в заголовке Authorization при запросах к защищённым маршрутам.

**Загрузка файлов:** хранение на диске сервера (папка uploads), доступ по URL; на клиенте используется FormData и multipart/form-data.

### 1.3 Требования к разработке

К системе предъявлены следующие требования.

**Функциональные:** регистрация и авторизация (email, пароль, JWT); восстановление пароля; профиль пользователя (имя, аватар, уникальный никнейм); CRUD мест с привязкой к категории и фотографиями; модерация мест (pending / approved / rejected) с причиной отклонения; комментарии к местам с возможностью прикрепления фото и модерацией; избранное; рейтинг мест (1–5); загрузка изображений; поиск по названию и по автору; фильтр по категории; разграничение прав (гость, пользователь, администратор); проверка дубликатов мест по названию.

**Нефункциональные:** адаптивный интерфейс (корректное отображение на мобильных устройствах); единый стиль и тёмная/светлая тема; защита API (JWT, проверка прав); ограничение размера и типа загружаемых файлов; возможность развёртывания на VPS (Node.js, PostgreSQL, Nginx).

### 1.4 Проектирование структуры данных

Спроектированы основные сущности БД.

**User** — пользователь (email, хэш пароля, имя, никнейм, аватар, флаг isAdmin).

**Category** — категория места (название).

**Place** — место (название, описание, адрес, категория, автор, статус модерации, причина отклонения).

**PlacePhoto** — фотография места (ссылка на файл, связь с местом).

**Comment** — комментарий (текст, место, автор, статус модерации, причина отклонения).

**CommentPhoto** — фотография в комментарии.

**Favorite** — избранное (пользователь, место).

**PlaceRating** — рейтинг места (пользователь, место, оценка 1–5).

Связи: у места один автор и одна категория; у места много фотографий и комментариев; у комментария — несколько фото; избранное и рейтинги связывают пользователей и места. При удалении места каскадно удаляются связанные фотографии, комментарии, записи из избранного и рейтингов.

## ГЛАВА 2. РАЗРАБОТКА ИНФОРМАЦИОННОЙ СИСТЕМЫ

### 2.1 Реализация серверной части

Серверная часть реализована на Node.js с использованием Express. Точка входа — файл index.js; подключение к PostgreSQL выполняется через Sequelize с конфигурацией из переменных окружения (DATABASE_URL). Подключены маршруты: auth (регистрация, вход, восстановление пароля, получение текущего пользователя), places (каталог, детали места, создание/редактирование/удаление мест, мои места), categories (список категорий), comments (комментарии к месту), favorites (добавление/удаление/список избранного), ratings (получение и установка рейтинга места), moderation (одобрение/отклонение мест и комментариев), uploads (загрузка изображений). Для защищённых маршрутов используется middleware проверки JWT и, при необходимости, проверка роли администратора. Реализованы проверка дубликатов мест по названию, ограничение размера и типа загружаемых файлов, выдача статики загруженных файлов по URL.

### 2.2 Реализация базы данных

Модели Sequelize определены в папке server/src/models: User, Category, Place, PlacePhoto, Comment, CommentPhoto, Favorite, PlaceRating. Связи заданы через hasMany, belongsTo; при удалении места выполняется каскадное удаление связанных записей. Миграции или синхронизация (в зависимости от выбранного подхода) создают таблицы в PostgreSQL. Индексы и ограничения уникальности (например, по email и никнейму пользователя) обеспечивают целостность данных.

### 2.3 Реализация клиентской части

Клиентское приложение построено на React (TypeScript) и Vite. Маршрутизация реализована с помощью React Router. Основные страницы: главная (PlacesList) с hero-секцией и блоком мест; каталог (Catalog) — «Все места» с фильтрами и поиском; избранное (Favorites); мои места (MyPlaces) с кнопкой добавления места; форма создания/редактирования места (PlaceForm); профиль (Profile); вход и регистрация (Login, Register); восстановление пароля (ForgotPassword, ResetPassword); панель модерации (Admin) — доступна только администратору. Компоненты: PlaceModal (карточка места с фото, описанием, рейтингом, комментариями, избранным), CommentsSection, StyledSelect (стилизованный выпадающий список), LotusIcon и рейтинг. Запросы к API выполняются через axios с подстановкой JWT в заголовок; токен и данные пользователя хранятся в localStorage. Реализованы адаптивная вёрстка и бургер-меню для мобильных устройств, тёмная и светлая тема.

### 2.4 Модерация и разграничение прав

Новые места и комментарии создаются со статусом «на модерации» (pending). В общем каталоге и в карточке места отображаются только одобренные (approved) места и комментарии. Автор видит свои места и комментарии независимо от статуса; при отклонении отображается причина. Администратор в панели модерации просматривает списки по статусам, одобряет или отклоняет контент (для отклонения обязательно указывается причина). Администратор может удалять одобренные места (в том числе созданные другими пользователями). Проверка прав выполняется на сервере по JWT и полю isAdmin в модели User.

### 2.5 Тестирование

Проведено ручное тестирование основных сценариев: регистрация и вход, восстановление пароля, создание и редактирование места, загрузка фотографий, добавление в избранное, выставление рейтинга, написание комментария с фото, одобрение и отклонение контента администратором, поиск по названию и автору, фильтр по категории, отображение на мобильном устройстве, переключение темы. Выявленные замечания устранены; система функционирует в соответствии с поставленными требованиями.
